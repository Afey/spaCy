spacy.en.EN¶

Tokenize English text, using a scheme that differs from the Penn Treebank 3 scheme in several important respects:

  • Whitespace is added as tokens, except for single spaces. e.g.,

    >>> [w.string for w in EN.tokenize(u'\nHello  \tThere')]
[u'\n', u'Hello', u' ', u'\t', u'There']

  • Contractions are normalized, e.g.

    >>> [w.string for w in EN.tokenize(u"isn't ain't won't he's")]
[u'is', u'not', u'are', u'not', u'will', u'not', u'he', u"__s"]

  • Hyphenated words are split, with the hyphen preserved, e.g.:

    >>> [w.string for w in EN.tokenize(u'New York-based')]
[u'New', u'York', u'-', u'based']

Other improvements:

  • Email addresses, URLs, European-formatted dates and other numeric entities not found in the PTB are tokenized correctly
  • Heuristic handling of word-final periods (PTB expects sentence boundary detection as a pre-process before tokenization.)

Take care to ensure your training and run-time data is tokenized according to the same scheme. Tokenization problems are a major cause of poor performance for NLP tools. If you’re using a pre-trained model, the spacy.ptb3 module provides a fully Penn Treebank 3-compliant tokenizer.

Tokenizer API¶

static EN.tokenize(self, unicode string) → Tokens

Tokenize a string.

The tokenization rules are defined in three places:

  • The data/<lang>/tokenization table, which handles special cases like contractions;
  • The data/<lang>/prefix file, used to build a regex to split off prefixes;
  • The data/<lang>/suffix file, used to build a regex to split off suffixes.
Parameters:string (unicode) – The string to be tokenized.
Returns:tokens – A Tokens object, giving access to a sequence of Lexemes.
Return type:Tokens

Lexeme Features Flag IDs¶

A number of boolean features are computed for English Lexemes. To access a feature, pass its ID to the spacy.word.Lexeme.check_flag() function.
